# Thai Public Procurement Intelligence
# Full Project Specification + Codex Prompt
# Prepared for: Personal portfolio project for AI Engineer / Data Engineer roles

================================================================================
1. PROJECT OVERVIEW
================================================================================

Project name:
Thai Public Procurement Intelligence

Short description:
An AI-powered search, analytics, and summarization platform for Thai public procurement open data. The system ingests public procurement datasets, normalizes and stores the data, provides searchable dashboards, supports semantic search, and uses a free/low-cost LLM provider such as Gemini free tier or OpenRouter free models to summarize procurement records and answer questions with evidence.

Primary portfolio goal:
Show practical ability in AI engineering and data engineering through an end-to-end deployed product:
- Data ingestion and cleaning
- Structured data modeling
- Full-stack application development
- Search and filtering
- Semantic retrieval / RAG
- LLM-based summarization
- Evidence-based AI assistant
- Production-minded architecture
- Free-tier deployment

Target users:
- Interviewers evaluating AI Engineer / Data Engineer candidates
- Developers exploring public procurement data
- Citizens or analysts who want to search Thai procurement records
- Portfolio reviewers who want a working demo

Key demo story:
"Paste or search procurement data, filter by province/agency/budget/category, open a procurement record, see normalized fields, view AI-generated summary, and ask questions like:
- What are the highest-budget IT procurement projects?
- Which agencies have many projects in a given category?
- Summarize this procurement notice in simple Thai.
- Find similar procurement records."

================================================================================
2. IMPORTANT CONSTRAINTS
================================================================================

This is a personal project and must be clearly separated from any employer/company work.

Cost constraints:
- Prefer free-tier services.
- Avoid paid APIs by default.
- Use Gemini free tier or OpenRouter free/free-tier models for generative AI.
- Cache LLM results to avoid repeated calls.
- Allow AI features to be disabled if no API key is configured.

Deployment constraints:
- Frontend should be deployable to Vercel free tier.
- Backend should be deployable to Render/Fly.io/Railway free or low-cost tier, or be containerized for local demo.
- Database should use Supabase free tier or local PostgreSQL with pgvector.
- The deployed demo should allow interviewers to try core features without needing private credentials.

UI constraints:
- Build custom UI with Next.js.
- Do not use Streamlit, Gradio, Retool, Appsmith, or other low-code dashboard tools.
- UI must look portfolio-ready.

Data constraints:
- Use publicly accessible procurement/open government datasets.
- Do not use confidential company data.
- The data source must be replaceable and documented.
- Build a fallback sample dataset seeder so the app works even if the public source changes.

LLM constraints:
- LLM usage should be optional.
- LLM provider abstraction must support Gemini and OpenRouter.
- Store generated summaries in DB so they are not regenerated repeatedly.
- Never send sensitive/private user data to LLM. This project should use public records only.

4. PUBLIC DATA SOURCE STRATEGY
================================================================================

Primary data source strategy:
Use Thai public procurement/open government data that can be downloaded as CSV, Excel, JSON, or queried through a public API.

Because public portals may change URLs or access rules over time, implement the ingestion system with a provider interface:
- data_sources/base.py
- data_sources/csv_importer.py
- data_sources/excel_importer.py
- data_sources/open_api_importer.py

Recommended source types to support:
1. CSV/Excel file import from open government/procurement datasets.
2. Manual admin upload of CSV/Excel for demo.
3. Seed dataset stored in the repository for offline demo.

Minimum dataset fields expected:
- project_id or generated hash id
- project_name
- agency_name
- province
- procurement_method
- procurement_type/category
- budget_amount
- winning_amount, if available
- winner_name, if available
- announcement_date
- contract_date, if available
- source_url, if available
- raw_text or description

Fallback sample data:
Create at least 100 synthetic-but-realistic sample records for development/demo if no public dataset is configured. Clearly label it as sample data in the UI.

Do not fabricate claims about real agencies when using sample data. Use obvious sample agency names such as:
- Sample Provincial Office A
- Sample Municipality B
- Sample Hospital C

5. FUNCTIONAL REQUIREMENTS
================================================================================

FR-001: Data ingestion
The system must import procurement records from CSV/Excel files and/or configured public URLs.

Acceptance criteria:
- User/developer can run ingestion from CLI.
- Ingestion validates required fields.
- Ingestion normalizes dates, budgets, province names, agency names.
- Duplicate records are detected using source id or content hash.
- Import result reports total inserted, updated, skipped, and failed rows.

FR-002: Data normalization
The system must normalize raw procurement records into a consistent schema.

Acceptance criteria:
- Budget values are converted into numeric decimal values.
- Dates are converted into ISO format.
- Province and agency text are trimmed and standardized.
- Missing optional fields are allowed.
- Invalid records are stored in ingestion_errors table or logged clearly.

FR-003: Procurement search
The UI must provide searchable procurement records.

Acceptance criteria:
- Search by keyword.
- Filter by province.
- Filter by agency.
- Filter by date range.
- Filter by budget range.
- Filter by procurement method/category.
- Sort by date, budget, and relevance.
- Pagination must be supported.

FR-004: Procurement detail page
Each procurement record must have a detail page.

Acceptance criteria:
- Shows normalized fields.
- Shows raw source text/description if available.
- Shows source URL if available.
- Shows AI summary if generated.
- Shows similar procurement records.
- Shows metadata such as imported_at and updated_at.

FR-005: AI summarization
The system must generate a concise summary for a procurement record using Gemini/OpenRouter.

Acceptance criteria:
- Summary is generated only when requested or during controlled batch processing.
- Summary is cached in DB.
- UI shows summary generation status.
- If LLM API is unavailable, UI displays a graceful fallback.
- Prompt instructs model to summarize only from provided text.

FR-006: Structured AI extraction
The system should optionally extract structured fields from raw text.

Acceptance criteria:
- Extracted fields are stored separately from original normalized fields.
- The UI must distinguish source fields from AI-extracted fields.
- Include confidence or notes when extraction is uncertain.
- Do not overwrite source-of-truth fields without explicit logic.

FR-007: Semantic search
The system must support semantic search over procurement names/descriptions/summaries.

Acceptance criteria:
- Records have embedding vectors.
- User can search with natural language queries.
- Results show similarity score or relevance explanation.
- Works even if AI answer generation is disabled.

FR-008: Hybrid search
The system should combine keyword and semantic search.

Acceptance criteria:
- Keyword-only search works.
- Semantic-only search works.
- Hybrid mode returns ranked results using both keyword relevance and vector similarity.
- The search mode is visible in the UI.

FR-009: AI Q&A assistant
The system must provide an AI assistant that answers questions based on retrieved procurement records.

Acceptance criteria:
- User asks a natural language question.
- Backend retrieves relevant records.
- LLM answers using only retrieved context.
- Answer includes cited procurement records.
- If context is insufficient, answer says it cannot determine.
- UI shows the records used as evidence.

FR-010: Analytics dashboard
The UI must include a dashboard summarizing dataset trends.

Acceptance criteria:
- Total records.
- Total budget.
- Records by province.
- Records by agency.
- Records by category/method.
- Monthly trend.
- Top budget projects.
- Top agencies by total budget.

FR-011: Similar records
The detail page must show similar procurement records.

Acceptance criteria:
- Uses vector similarity or fallback keyword matching.
- Shows 5 to 10 similar records.
- Avoids showing the same record as itself.

FR-012: Dataset management page
A simple admin/developer page or CLI must show ingestion status.

Acceptance criteria:
- Last import time.
- Number of records.
- Number of records with embeddings.
- Number of records with summaries.
- Error count.
- Optionally trigger import in local/dev mode only.

FR-013: Public demo mode
The deployed app must be usable by interviewers.

Acceptance criteria:
- Demo has seeded data.
- Demo does not require login for basic search/dashboard/detail.
- AI calls may be rate-limited.
- Demo includes a visible note about data source and limitations.

FR-014: API documentation
Backend must expose OpenAPI docs.

Acceptance criteria:
- FastAPI docs available at /docs in non-production or demo mode.
- Main endpoints are documented with request/response schemas.

FR-015: Export
The system should allow export of search results.

Acceptance criteria:
- Export current filtered results as CSV.
- Respect pagination or allow export limit.
- Include normalized fields, not embeddings.

6. NON-FUNCTIONAL REQUIREMENTS
================================================================================

NFR-001: Cost control
The system must minimize recurring cost.

Requirements:
- Cache AI summaries.
- Cache embeddings.
- Avoid regenerating on every page load.
- Use free-tier friendly DB size.
- Include environment variable to disable LLM features.
- Provide local-only batch scripts for expensive processing.

NFR-002: Performance
The app must feel responsive for demo use.

Requirements:
- Search response target under 1.5 seconds for up to 50k records.
- Use DB indexes for filters.
- Use pagination.
- Use async backend endpoints where useful.
- Avoid loading all records into memory.

NFR-003: Reliability
The system must handle imperfect public data.

Requirements:
- Bad rows must not crash full import.
- Log failed rows.
- Retry network fetches with backoff.
- Validate schemas before DB insert.
- Provide clear error messages.

NFR-004: Security
The deployed demo must not expose secrets or allow abuse.

Requirements:
- Store API keys only in environment variables.
- Never expose API keys to frontend.
- Rate limit AI endpoints.
- Validate upload file size/type.
- Sanitize user inputs.
- Avoid executing arbitrary code.
- CORS must be restricted to frontend domain in production.

NFR-005: Maintainability
The codebase must be easy to understand.

Requirements:
- Clear folder structure.
- Type hints in Python.
- TypeScript types in frontend.
- Pydantic schemas for API.
- DB migrations.
- README with setup and architecture.
- Comments for non-obvious logic.

NFR-006: Observability
The app must provide basic operational visibility.

Requirements:
- Structured logs for ingestion and AI calls.
- API request logging.
- Ingestion status endpoint.
- Error table or log file for import failures.
- Optional metrics endpoint if simple to add.

NFR-007: Accessibility
The UI should be usable and readable.

Requirements:
- Responsive layout.
- Keyboard-friendly controls.
- Sufficient color contrast.
- Loading/error/empty states.
- Clear Thai/English text labels.

NFR-008: Explainability
AI outputs must be transparent.

Requirements:
- Show source records used for answers.
- Show "AI-generated" label for summaries.
- Show when answer is uncertain.
- Avoid making claims not supported by records.

NFR-009: Portability
The project must run locally.

Requirements:
- docker-compose up should start DB and backend dependencies.
- Frontend and backend setup documented.
- Seed data script included.
- .env.example files included.

NFR-010: Separation from employer work
The project must not contain employer code, employer data, or internal architecture.

Requirements:
- Use public data only.
- Use original code.
- Do not mention employer systems in README except resume context outside the project.

12. SEARCH AND RAG DESIGN
================================================================================

Record text for embedding:
Combine these fields:
- project_name
- agency_name
- province
- procurement_method
- procurement_category
- budget_amount
- announcement_date
- raw_text or normalized_text
- AI summary if available

Embedding process:
1. Select records without embeddings.
2. Build embedded_text.
3. Generate embedding using configured provider.
4. Store in procurement_embeddings.
5. Re-run only if text hash changes.

Retrieval flow for AI assistant:
1. Receive question and optional filters.
2. Run hybrid search:
   - PostgreSQL full-text search
   - vector similarity search
3. Merge and rank results.
4. Select top 5-8 records.
5. Build compact context.
6. Ask LLM to answer using context only.
7. Return answer and cited records.

Hybrid ranking idea:
final_score = 0.55 * keyword_score + 0.45 * vector_score

Fallback:
If vector search is unavailable, use keyword search only.
If LLM is unavailable, show retrieved records but no generated answer.

3. Map raw columns to canonical fields.

5. Validate required fields.
6. Generate content_hash.
7. Upsert procurement record.
8. Log row errors.
9. Update ingestion_run counts.

Deduplication:
Priority:
1. source_record_id if available
2. content_hash from project_name + agency_name + date + budget
3. fuzzy matching optional later
